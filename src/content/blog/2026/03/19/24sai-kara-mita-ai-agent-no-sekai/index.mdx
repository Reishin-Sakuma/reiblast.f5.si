## そもそもAIエージェントって何？

まず「[AIエージェント](https://ja.wikipedia.org/wiki/%E3%82%BD%E3%83%95%E3%83%88%E3%82%A6%E3%82%A7%E3%82%A2%E3%82%A8%E3%83%BC%E3%82%B8%E3%82%A7%E3%83%B3%E3%83%88)」って何なのかを整理します。このブログの通例なんで書きます。笑

今まで紹介してきたChatGPTやClaudeは、基本的に**1問1答**なんですよね。

「これ教えて」→「こうだよ」。「このコード書いて」→「はいどうぞ」。

AIエージェントは違います。

<mark style="background-color:#ffcc00"><strong>「ゴールを伝えたら、自分で考えて、自分で手を動かして、完了まで持っていく」</strong></mark>

これがAIエージェントです。

たとえば「このリポジトリにログイン機能を追加して」って言ったら、コードを読んで、設計を考えて、ファイルを作って、テストを書いて、動作確認までやってくれる。途中で「ここどうしますか？」って聞いてくることもあるけど、基本的には**自律的に動く**。

<span style="color:orange">**1問1答のチャットAI → 自分で考えて動くAIエージェント**</span>

この進化、地味に見えるかもしれないけど、使ってみると衝撃でした。

## 2025年が「AIエージェント元年」だった

2025年あたりから「AIエージェント」って言葉がIT業界でめちゃくちゃバズり始めたんですよね。

各社がこぞってエージェント機能をリリースして、2025年後半には実際に業務で使えるレベルのツールが出揃ってきました。

そして2026年の今、もう**本格稼働フェーズ**に入ってます。

<span style="color:#8f20ff">「え、そんなに急に？」って感じですけど、ITの世界って1年あれば景色が変わるんですよね。。。</span>

## 僕が実際に使ったツール

ここからは実際に触ってみた話をします。

### Claude Code ― メインで半年使い続けたやつ

一番長く使ってるのが[Claude Code](https://docs.anthropic.com/ja/docs/claude-code)です。ターミナル上で動くAIエージェントで、コードの読み書き・ファイル操作・コマンド実行を**自律的にやってくれる**。

最初に触ったときの正直な感想は「え、これ普通に怖いな」でした。笑

「このスクリプトのエラーを直して」って頼んだら、ファイルを勝手に読んで、原因を特定して、修正して、動作確認のコマンドまで実行してくれる。ここまでを**自分で判断して順番にこなす**んですよ。

<span style="color:#0000ff">実はこのブログサイト自体、Claude Codeを使いながら作りました。</span>Astroの設定をいじったり、コンポーネントを作ったり、スクリプトのデバッグをしたり。記事を書いたあとの「OGP画像を生成して」「ビルドして確認して」「エラー出たら直して」みたいな作業も全部任せられて、エラーが出ても自分でデバッグして完了まで持っていってくれる。

あと地味に便利なのが<span style="color:#ff7f00">**ブログ記事のファクトチェック**</span>。書き上げたあとに「この記事の技術的な説明に誤りがないか確認して」って渡すと、怪しい部分を指摘してくれる。もちろん最終判断は自分でしますが、見直しの精度が上がりました。

**半年使い続けて特に効いてる使い方**：

- <span style="color:#ff7f00">**IaC（[Ansible](https://ja.wikipedia.org/wiki/Ansible_%28%E3%82%BD%E3%83%95%E3%83%88%E3%82%A6%E3%82%A7%E3%82%A2%29)のタスクファイル、シェルスクリプト）の初稿生成**</span> — 仕様を伝えると叩き台を作ってくれる。ゼロから書くより圧倒的に早い
- <span style="color:#ff7f00">**既存スクリプトの[リファクタリング](https://ja.wikipedia.org/wiki/%E3%83%AA%E3%83%95%E3%82%A1%E3%82%AF%E3%82%BF%E3%83%AA%E3%83%B3%E3%82%B0)**</span> — 「このスクリプト整理して」で全体を見てまとめてくれる
- <span style="color:#ff7f00">**障害ログの解析補助**</span> — ログをそのまま渡して「何が起きてるか教えて」で概要をまとめてくれる

もちろん最終確認は自分でやります。**「Claude Codeが言ってたから大丈夫」は絶対NG**。でも、確認する余裕が生まれるくらいには作業スピードが変わりました。


## インフラエンジニアとAIエージェント

さて、ここからが本題です。インフラエンジニアの僕から見た話。

### IaC（Infrastructure as Code）との組み合わせ

3〜4年目の振り返り記事で「サーバ構築の自動化」について書きましたが、あの頃は[Ansible](https://ja.wikipedia.org/wiki/Ansible_(%E3%82%BD%E3%83%95%E3%83%88%E3%82%A6%E3%82%A7%E3%82%A2))のPlaybookを自分でゼロから書いてました。

今は[IaC](https://ja.wikipedia.org/wiki/Infrastructure_as_Code)のコードをAIエージェントに初稿を作ってもらって、人間がレビューして、問題なければ適用する、みたいなフローが現実的になってきてます。

<span style="color:orange">**AIがコードを書く → 人間がレビューする → 人間が適用する**</span>

この流れ、[コードレビュー](https://ja.wikipedia.org/wiki/%E3%82%B3%E3%83%BC%E3%83%89%E3%83%AC%E3%83%93%E3%83%A5%E3%83%BC)の文化がある現場なら自然に導入できるんですよね。

### AIOps：障害対応の自動化

インフラエンジニアにとってもう一つ大きいのが[AIOps](https://en.wikipedia.org/wiki/AIOps)です。

障害が起きたときの対応って、だいたいこういう流れじゃないですか：

1. アラートが飛ぶ
2. ログを見に行く
3. 原因を特定する
4. 暫定対処する
5. 報告書を書く

<span style="color:#ff0000">これ、夜間作業の最中に障害が出たりするとマジでしんどい。</span>作業を止めて原因切り分けからやり直し、ってなったときの絶望感、わかる人にはわかると思います。笑

AIエージェントが進化すると、このうちの2〜4あたりを**自動でやってくれる**可能性がある。

アラートを検知して、関連するログを集めて解析して、「たぶんこれが原因です、暫定対処としてこのサービスを再起動しました」みたいなレポートが上がってくる世界。

<span style="color:#8f20ff">まだ完全には実現してないですけど、部分的にはもう動き始めてるんですよね。</span>

### Platform Engineering との連携

最近話題の[Platform Engineering](https://en.wikipedia.org/wiki/Platform_engineering)とも相性がいいです。

開発者が使う社内プラットフォームに、AIエージェントを組み込んで、「環境構築して」「デプロイして」みたいな依頼をチャットで受けて自動実行する。

インフラチームが全部手動で対応しなくてよくなる、という話です。

<span style="color:#0000ff">これ実現したら、インフラエンジニアの働き方もかなり変わるだろうな。</span>

## ただし、注意点は山ほどある

ここまで良いことばかり書いてきましたが、注意点もちゃんと書きます。「生成AIを仕事で使う前に知っておきたいこと」で書いた内容に加えて、エージェント特有のリスクがあるので。

<Note type="warning" title="AIエージェント特有のリスク">

**間違った方向で走り続ける**

チャットAIなら1回の回答で終わるので、間違ってても被害は小さい。でもエージェントは「自分で考えて動く」ので、**間違った方向に全力で突っ走ることがある**。気づいたときには大量のファイルが変わってた、みたいなことが普通に起きます。

**本番環境へのアクセス権限は絶対に与えない**

これは声を大にして言いたい。AIエージェントに本番サーバの認証情報を渡すのは<span style="color:#ff0000">**絶対NG**</span>です。検証環境で試して、問題ないことを確認してから人間が本番に適用する。このフローは崩しちゃダメです。

**変更内容は必ず人間がレビューする**

エージェントが「完了しました」と言っても、何がどう変わったのかを人間の目で確認すること。<strong>「AIがやったから大丈夫でしょ」は一番危ない思い込み</strong>です。

</Note>

<span style="color:#8f20ff">前にも書きましたけど、「AIが言ってたから」は言い訳にならないんですよね。エージェントになっても、最終責任は人間にあります。</span>

## まとめ：AIエージェントは「次のステージ」

ここまで読んでくれた方、ありがとうございます。

正直、AIエージェントの話って2026年3月現在でもまだ発展途上で、「これが正解」みたいなものは確立されてないです。

でも確実に言えるのは、<mark style="background-color:#ffcc00"><strong>チャットAIの「次」が来ている</strong></mark>ということ。

1問1答のAIを使いこなせるようになった人が、次に覚えるべきはエージェントとの協業だと思ってます。

<span style="color:orange"><strong>「AIに聞く」から「AIと一緒に作る」へ。</strong></span>

インフラエンジニアとして、この変化にちゃんとついていきたいなと。IaCの自動化をやってきた経験があるからこそ、エージェントとの相性の良さは肌で感じてます。
